# Rosty PRD

## 1. 문서 목적

이 문서는 웨딩홀 운영진과 근무자를 위한 모바일 스케줄 관리 앱 `Rosty`의 제품 요구사항을 정의한다.
현재 기준은 React Native + Expo Router + Supabase 기반 모바일 앱이며, 구현과 설계의 기준 문서로 사용한다.

## 2. 제품 한 줄 정의

Rosty는 웨딩홀 운영진이 일정 생성, 신청 집계, 인원 배정, 취소 처리, 근무 기록, 예상 급여 확인까지 한 앱에서 관리하고, 근무자는 본인 일정과 배정 상태를 빠르게 확인하고 대응할 수 있게 해주는 모바일 운영 도구다.

## 3. 문제 정의

현재 웨딩홀 운영 현장에서는 다음 문제가 반복된다.

- 일정 공지와 신청이 메신저에 흩어져 누락과 중복 확인이 잦다.
- 운영진마다 배정 기준이 달라 공정성과 재현성이 떨어진다.
- 취소 요청과 공석 대응이 실시간으로 추적되지 않는다.
- 실제 근무 이력과 예상 급여를 한 화면에서 확인하기 어렵다.
- 여러 웨딩홀을 함께 운영할 때 권한과 데이터 경계가 불명확해진다.

## 4. 제품 목표

### 4.1 비즈니스 목표

- 웨딩홀 운영의 일정 커뮤니케이션을 메신저 중심에서 앱 중심으로 전환한다.
- 신청, 배정, 취소, 근무 기록의 흐름을 표준화한다.
- 홀 단위 데이터와 권한을 분리해 확장 가능한 운영 구조를 만든다.
- 운영진이 일정 배정과 변경 처리에 쓰는 시간을 줄인다.

### 4.2 사용자 목표

- 근무자는 가능한 일정만 빠르게 제출하고 확정 결과를 놓치지 않아야 한다.
- 운영진은 일정별 인력 현황과 추천 기준을 보고 빠르게 배정을 완료해야 한다.
- Admin은 멤버, 역할, 급여 기준을 홀 단위로 안전하게 관리해야 한다.
- Super Admin은 플랫폼 전체 홀 승인과 운영 상태를 통제할 수 있어야 한다.

## 5. 비목표

- 실제 급여 정산, 세금 계산, 이체 처리
- 데스크톱 웹 제품 제공
- 외부 ERP/HR 시스템 연동
- 웨딩홀 간 인력 공유 마켓
- 채팅 기능 자체 구현

## 6. 대상 사용자와 역할

### Employee

- 본인 가능 여부 제출 및 수정
- 본인 배정 결과 확인
- 본인 취소 요청 제출
- 본인 근무 기록 확인 또는 입력
- 본인 예상 급여 조회

### Manager

- 일정 생성 및 수정
- 신청 현황 확인
- 배정 추천 검토 및 수동 배정
- 취소 요청 승인 또는 거절
- 공석 상황 대응

### Admin

- Manager 권한 포함
- 멤버 관리
- 홀 내 역할 관리
- 시급 및 급여 정책 관리
- 초대 링크 관리

### Super Admin

- 웨딩홀 등록 요청 승인 또는 거절
- 플랫폼 운영 상태 관리
- 정책성 운영 이슈 대응

## 7. 핵심 제품 원칙

- 모바일 우선 UX를 유지한다.
- 모든 핵심 데이터는 홀 단위로 분리한다.
- 역할 기반 접근 제어를 제품과 백엔드 모두에서 강제한다.
- 상태 변경은 이력 추적이 가능해야 한다.
- 알림은 단순 공지가 아니라 액션 가능한 흐름의 일부여야 한다.

## 8. 핵심 사용자 시나리오

### 8.1 근무자 시나리오

1. 사용자가 소셜 로그인으로 앱에 가입한다.
2. 프로필을 입력하고 소속 홀에 참여한다.
3. 일정 목록에서 가능한 일정에 가능/불가를 제출한다.
4. 배정 확정 후 본인 배정 결과를 확인한다.
5. 필요 시 취소 요청을 제출한다.
6. 근무 후 실제 시간을 확인하고 예상 급여를 본다.

### 8.2 운영진 시나리오

1. 운영진이 웨딩 일정과 슬롯을 생성한다.
2. 신청자, 미응답자, 마감 상태를 확인한다.
3. 추천 결과를 참고해 인원을 배정하고 확정한다.
4. 취소 요청과 공석 발생을 처리한다.
5. 근무 기록과 예상 급여 기준을 확인한다.

### 8.3 관리자 시나리오

1. Admin이 멤버를 초대하거나 역할을 변경한다.
2. 홀별 급여 정책과 직원 시급을 관리한다.
3. 홀 단위 설정과 운영 기준을 유지한다.

## 9. 기능 요구사항

### 9.1 인증 및 온보딩

- 사용자는 소셜 로그인으로 가입한다.
- 초기 지원 provider 기본 우선순위는 `Google -> Kakao -> Naver`로 둔다.
- 최초 로그인 후 이름, 전화번호, 성별을 입력한다.
- 초대 링크로 진입한 경우 홀 소속을 자동 연결할 수 있어야 한다.
- 초대 링크가 없는 사용자는 홀 생성 요청 또는 승인 대기 흐름으로 이동한다.

### 9.2 홀과 권한

- 한 사용자는 여러 홀에 소속될 수 있다.
- 역할은 홀마다 독립적으로 관리된다.
- 앱 상단 또는 전용 화면에서 현재 작업 홀을 전환할 수 있어야 한다.
- 모든 조회와 변경은 현재 홀 또는 사용자의 membership 기준으로 제한한다.

### 9.3 일정 생성

- Manager 이상 권한은 일정 생성이 가능하다.
- 일정 생성 시 날짜, 예식 수, 기본 메모, 신청 마감 시각을 입력한다.
- 일정 생성 시 표준 포지션 슬롯을 자동 생성할 수 있어야 한다.
- 생성된 일정은 목록과 상세 화면에서 확인 가능해야 한다.

### 9.4 가능 여부 제출

- Employee는 일정별로 가능/불가/미응답 상태를 가진다.
- 마감 전까지 제출값 수정이 가능해야 한다.
- 운영진은 신청자, 미응답자, 불가자를 구분해 확인할 수 있어야 한다.

### 9.5 배정

- 운영진은 일정 슬롯별로 인원을 배정할 수 있어야 한다.
- 성별 제한, 포지션 이력, 최근 배정, 월간 신청 횟수 같은 기준을 추천 근거로 제공한다.
- 자동 추천 결과는 참고용이며 운영진이 최종 수정할 수 있어야 한다.
- 외부 게스트 인력도 이름만으로 임시 배정할 수 있어야 한다.
- 배정 확정 시 관련 사용자에게 알림을 보내야 한다.

### 9.6 취소 요청

- Employee는 본인 확정 배정에 대해 취소 요청을 보낼 수 있다.
- 요청에는 사유를 입력해야 한다.
- Manager 이상은 승인 또는 거절할 수 있어야 한다.
- 승인 시 해당 배정은 공석 상태로 전환된다.
- 처리 결과는 요청자와 관련 운영진에게 알림으로 전달된다.

### 9.7 근무 기록

- 근무 시작/종료 시간을 기록할 수 있어야 한다.
- 입력 주체는 Employee 또는 운영진으로 둘 수 있다.
- 수정 시 마지막 수정자와 시간을 남겨야 한다.

### 9.8 예상 급여

- 홀별 초과근무 기준 시간과 배율을 설정할 수 있어야 한다.
- 직원별 시급을 관리할 수 있어야 한다.
- 예상 급여는 근무 시간, 시급, 초과근무 기준으로 계산한다.
- 이 기능은 참고용이며 정산 완료 상태를 대체하지 않는다.

### 9.9 알림

- 인앱 알림과 푸시 알림을 함께 지원한다.
- 일정 생성, 신청 마감 임박, 배정 확정, 취소 처리, 홀 승인 상태 변경을 알린다.
- 알림은 관련 상세 화면으로 연결되어야 한다.

### 9.10 이력과 감사 가능성

- 배정 변경, 취소 처리, 근무 기록 수정은 추적 가능한 상태여야 한다.
- 최소한 변경 시각과 변경 주체를 남겨야 한다.

## 11. 출시 우선순위

### Phase 1

- 소셜 로그인 진입
- 프로필 작성
- 홀 소속 및 현재 홀 선택
- 일정 목록/상세
- 가능 여부 제출

### Phase 2

- 일정 생성
- 신청 현황 조회
- 수동 배정
- 배정 확정

### Phase 3

- 취소 요청/처리
- 근무 기록
- 예상 급여
- 인앱 알림/푸시 알림

## 12. 성공 지표

- 일정 가능 여부 제출률
- 배정 확정 후 확인률
- 일정 생성부터 배정 완료까지 걸리는 평균 시간
- 취소 요청 처리 시간
- 미응답 인원 비율
- 일정/배정/알림 누락 건수

## 13. 기술/운영 전제

- 프론트엔드는 React Native + Expo Router 기반 모바일 앱으로 개발한다.
- 백엔드는 Supabase Auth + Postgres + RLS를 기본 전제로 한다.
- Android와 iOS를 모두 지원 대상으로 삼되, 현재 개발 검증은 Android 우선이다.
- 비밀값은 저장소가 아닌 로컬/CI/EAS/Supabase secret 체계로 관리한다.

## 14. 오픈 이슈

- 초대 링크의 만료 정책과 재발급 정책
- 배정 추천 점수의 상세 계산식
- 근무 기록 수정 권한의 세부 정책
- 푸시 알림 묶음 발송과 실시간 발송 기준
- 홀 생성 요청 승인 프로세스의 운영 기준
