# 근무 도메인 정의

## 목적

이 문서는 `근무`, `근무 신청`, `근무 배정`, `포지션`의 의미를 고정하는 기준 문서다. 이후 타입, API, 화면 흐름, 데이터 모델은 이 문서의 개념 정의를 우선 참조한다.

이 문서는 구현 상세나 이행 계획을 다루지 않는다. 오직 "무엇이 정답 개념인가"만 정의한다.

## 핵심 용어

### 근무

`근무`는 관리자가 먼저 생성하는 기준 일정이다.

- 근무는 특정 날짜에 속한다.
- 근무는 시작 시각과 종료 시각을 가진다.
- Release 1에서는 시스템 전체 기준 하루에 1개만 존재한다.
- 근무는 시간 정보를 소유한다.

직원은 근무를 직접 만들지 않는다. 관리자가 생성한 근무를 기준으로 신청과 배정이 이어진다.

### 근무 신청

`근무 신청`은 직원이 특정 근무에 참여 가능함을 표시하는 행위이자 기록이다.

- 신청은 특정 근무를 대상으로 한다.
- 신청은 특정 직원과 연결된다.
- Release 1에서 신청 단계의 의미는 `가능 여부 표시`다.
- 신청 단계에서는 포지션을 선택하지 않는다.
- 신청은 근무 시간을 바꾸지 않는다.

Release 1에서는 `신청 = 참여 가능`을 뜻한다. 신청하지 않은 상태는 별도의 명시적 불가 응답으로 보지 않고, 미신청 또는 미응답으로 본다.

### 근무 배정

`근무 배정`은 관리자가 신청자 중 일부를 실제 근무자로 확정하는 행위이자 기록이다.

- 배정은 특정 근무를 대상으로 한다.
- 배정은 특정 직원과 연결된다.
- 배정은 포지션을 포함한다.
- 배정은 처리 시각과 처리 담당자를 가진다.
- 직원이 실제로 보게 되는 "배정된 근무"는 신청이 아니라 배정 결과다.

배정은 신청을 전제로 한다. 관리자는 신청자 목록을 보고 누가 어떤 포지션으로 들어갈지 결정한다.

### 포지션

`포지션`은 실제 현장 배치 단위다. 시간 정보의 일부가 아니라, 배정 시점에 확정되는 역할 단위다.

이 프로젝트의 canonical 포지션 목록은 PRD 기준으로 다음과 같다.

- 팀장
- 스캔
- 메인
- 드레스
- 대기실
- 축가
- 매니저
- 안내
- 드레스실

포지션은 근무가 아니라 배정이 소유한다. 즉, 시간은 근무에 속하고 포지션은 배정에 속한다.

## 엔티티 정의

### Work

`Work`는 관리자가 생성하는 근무 일정이다.

최소 개념 속성:

- 날짜
- 시작 시각
- 종료 시각

Release 1에서는 포지션별 필요 인원을 근무 생성 시점에 확정하지 않는다.

### WorkApplication

`WorkApplication`은 직원이 특정 근무에 참여 가능함을 제출한 기록이다.

최소 개념 속성:

- 대상 근무
- 신청 직원
- 신청 시각
- 참여 가능 여부

Release 1에서는 포지션 희망, 시간대 제안, 역할 선택을 신청에 포함하지 않는다.

### WorkAssignment

`WorkAssignment`는 관리자가 특정 근무에 특정 직원을 특정 포지션으로 확정한 기록이다.

최소 개념 속성:

- 대상 근무
- 배정 직원
- 배정 포지션
- 배정 시각
- 배정 담당자

필요 시 실제 배치 위치나 근무 장소를 함께 가진다.

## 관계와 제약

### 기본 관계

- 근무 1개에는 여러 신청이 연결될 수 있다.
- 신청은 근무와 직원을 연결한다.
- 근무 1개에는 여러 배정이 연결될 수 있다.
- 배정은 근무와 직원을 연결하고 포지션을 부여한다.
- 배정은 신청을 전제로 한다.

### Release 1 제약

- 시스템 전체 기준 하루에 근무는 1개만 존재한다.
- 직원은 생성된 근무에 대해서만 신청할 수 있다.
- 직원 신청은 가능 여부만 제출한다.
- 포지션은 신청 단계가 아니라 배정 단계에서만 확정한다.
- 포지션별 필요 인원은 배정 단계에서 판단한다.

## 설계 배경

### 왜 근무를 먼저 생성해야 하는가

현재 제품 흐름은 관리자가 운영 일정을 먼저 만들고, 직원이 그 일정에 참여 가능한지를 표현한 뒤, 관리자가 신청자 중에서 실제 인원을 배치하는 구조다. 따라서 `근무`가 신청보다 먼저 존재해야 이후 개념이 자연스럽게 정리된다.

### 왜 신청과 배정을 분리해야 하는가

신청은 직원의 가능 여부 표현이고, 배정은 관리자의 최종 결정이다. 두 개념을 하나의 레코드로 합치면 "직원이 무엇을 제안했는지"와 "관리자가 무엇을 확정했는지"가 섞여 책임 경계가 흐려진다.

### 왜 시간과 포지션을 분리해야 하는가

근무 시간은 운영 일정의 속성이고, 포지션은 인력 배치 결과의 속성이다. 시간을 포지션과 같은 층위로 두면 배정 구조가 불명확해진다. 이 프로젝트에서는 `시간 = 근무`, `포지션 = 배정` 원칙을 유지한다.

### 왜 PRD의 포지션 목록을 기준으로 삼는가

근무 도메인의 실제 운영 단위는 현장 포지션이다. 임시로 단순화된 역할 체계를 계속 쓰면 이후 배정, 급여, 출퇴근, 운영 대시보드와 연결할 때 다시 개념을 뒤집게 된다. 따라서 포지션 taxonomy는 PRD를 기준으로 고정한다.

## 참조 규칙

- 근무 관련 새 문서를 작성할 때는 이 문서의 용어를 그대로 사용한다.
- 구현 세부 설계가 이 문서와 충돌하면 이 문서를 우선 검토하고, 필요 시 먼저 문서 정의를 갱신한다.
- PRD는 기능 범위를 설명하고, 이 문서는 도메인 개념을 설명한다.
